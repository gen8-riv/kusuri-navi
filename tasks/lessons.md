# Lessons

- 2026-03-08: sticky UI を複数段で積む画面では `top` を固定値にしない。実寸から CSS 変数を更新し、関連要素で共有する。
- 2026-03-08: 複数ページで同じクラス名を使う場合、状態変更の selector は必ずコンテナ単位で絞る。
- 2026-03-08: 単一 HTML でも、長い HTML 文字列生成はセクション単位で関数分割しておくと後続のレイアウト修正が速い。
- 2026-03-08: 医療系のUIで親しみやすさを出す場合も、視認性が最優先。固定列やstickyヘッダーには色差だけでなく影や境界も併用する。
- 2026-03-08: 凡例がある評価軸は、詳細展開の中だけに閉じ込めない。閉じた状態でも最低限の判断材料を出す。
- 2026-03-08: フィルター用の表記は、業務上意味のある粒度に正規化する。識別用の補助情報はカード側で残す。
- 2026-03-08: 医薬品データを増やす前に、まず剤形がアプリ対象に合っているかを棚卸しする。貼付剤や内用液は味ナビの対象から外す。
- 2026-03-08: 薬効分類の誤りも医療リスクになる。仮置きカテゴリは残さず、誤解のない分類へ修正する。
- 2026-03-08: 監修前提の医療データでは、公開判定を各画面で個別実装しない。`isPublicMed` のような単一ゲートに集約して漏れを防ぐ。
- 2026-03-08: 安全上の理由で一覧を空にしうるUIでは、空状態に理由を必ず明記する。説明がないと不具合に見える。
- 2026-03-08: 回帰チェックは汎用語句の有無ではなく、対象UIの selector や関数名で判定する。別機能の文言追加で壊れやすい。
- 2026-03-08: 医療系MVPのリスクを下げる最短手は、独自知見を増やすことではなく、公的ソースの範囲に公開対象を絞ること。
- 2026-03-08: 公的資料ベースの設計でも、未掲載薬への推測補完や同成分の自動横展開を入れた瞬間にリスクが跳ねる。公開範囲は資料の粒度に合わせる。
- 2026-03-08: 公的PDFの表を実装へ落とすときは、まず `officialSourceMap` のような明示的な行列データを作る。既存の自由記述から逆算しない。
- 2026-03-08: macOS 環境なら PDFKit で公式PDFの本文抽出ができる。医療系データの棚卸しでは外部OCRより先に試す。
- 2026-03-08: 公開対象を絞ったアプリでは、フィルターUIも固定で置かない。公開データから動的生成しないと、未掲載カテゴリが残って誤解を招く。
- 2026-03-08: `official / reviewed / internal` のような層は配列や関数で分けておく。単一の `getPublicMedicines()` だけではレビュー画面や運用判断が曖昧になる。
- 2026-03-08: 内部候補が多いレビュー画面は単一テーブルにしない。公開レイヤーと内部候補をセクション分割した方が優先順位を判断しやすい。
- 2026-03-08: 画面表示とCSV出力で分類ロジックを別々に持たない。`row.layer` のような共通フィールドを起点にするとズレにくい。
- 2026-03-08: ユーザーが「入力経路は1つでよい」と示したら、運用設計でも正本を1つに寄せる。CSVや別経路は監査・出力用途に下げる。
- 2026-03-08: 静的HTMLで監修ワークフローを載せる場合、`verified` だけで公開判定しない。`published` のような最終ゲートを別に持たせる。
- 2026-03-08: ブラウザ保存の管理UIでは、別タブ競合を前提に `version` と `storage` イベントを入れる。入力フォームだけ増やすと誤上書きが起きやすい。
- 2026-03-08: 主観項目を監修画面に足すときは、別テーブルや別JSONを作らず `review` を正本にする。公開UI側で別ソースを読むと監修状態と内容がズレる。
- 2026-03-08: 相性は自由記述のおすすめ食品リストより、固定列のマトリクス入力へ正規化した方が再利用しやすい。カード・表・CSVへ同じ判定を流せる。
- 2026-03-08: 味表現を監修入力で扱うなら、複数チェックボックスより `主な味 / 副次的な味 / 後味` に正規化した方がぶれにくい。公開UI用の味タグはそこから自動生成する。
- 2026-03-08: 香りも自由入力を正本にしない。`主な香り / 副次的な香り / 詳細` に分け、表示用ラベルはそこから合成した方が監査しやすい。
- 2026-03-08: 同じ薬詳細を一覧カードと表ポップアップで見せる場合、詳細セクションは `renderDetailSections()` のように共通化する。片側だけ直すとメモ欄や凡例位置がすぐズレる。
- 2026-03-08: 一覧と表の件数差を防ぐには、公開判定だけでなくカテゴリ絞り込みも共通関数に寄せる。件数表示を各画面に出すと、将来の差分も発見しやすい。
- 2026-03-08: 件数ラベルは単に `18件` と出さない。公開母数と絞り込み件数を分けて表示しないと、検索欄の復元や一時的な絞り込みで不具合に見えやすい。
- 2026-03-08: 単一ページでタブ切替しているUIは、ページ切替時にスクロール位置を明示的にリセットする。sticky ヘッダーがあると、先頭要素が欠落したように見えやすい。
- 2026-03-08: 高さのある表ヘッダーに縦 sticky を掛けると、先頭2行前後を覆ってしまうことがある。実ブラウザで欠落に見えるなら、まず sticky を疑い、必要なら縦 sticky を外す。
- 2026-03-08: タブごとに期待する作業が違うなら、カードの初期状態も変える。保存・確認用タブは最初から展開して、編集欄を直接触れる方が速い。
- 2026-03-08: お気に入り状態をカードとポップアップで共有するなら、DOM 構造依存の selector ではなく `data-fav-id` のような共通キーで同期する。
- 2026-03-08: `メモ` と `お気に入り` のように保存系の目的が近いタブは分けすぎない。上位タブは1つにまとめ、内部フィルターで切り替えた方が役割が明確になる。
- 2026-03-08: 保存用ページでは、詳細カードを常時開くより「カードは閉じる / メモだけ外出しで常時編集可」の方が一覧性と編集速度の両立がしやすい。
- 2026-03-08: 表に保存系の列を足すなら、`商品名` の近くに置く。ハートやメモが右端にあると、スマホでは相性評価より先に横スクロールが必要になって使いにくい。
- 2026-03-08: 保存済みや収集系のタブはハートより星の方が中立で、行内のお気に入りハートと役割が混同しにくい。
- 2026-03-08: iPhone Safari では、横スクロール領域内の `thead th` に直接 sticky を掛けても安定しないことがある。表ヘッダー固定が必要なら、表示用ヘッダーを別ラッパーに分離して `scrollLeft` を同期する方が堅い。
- 2026-03-08: クローンした固定ヘッダーは「表示」と「レイアウト確保」が二重になりやすい。元 `thead` を高さ確保に使う場合は、その高さぶんを相殺しないと1行目の上に白い空白が残る。
- 2026-03-08: 別ラッパー式の固定ヘッダーは、`table.scrollWidth` だけ合わせても横向きで列幅がずれることがある。各ヘッダーセルの実測幅をクローン側へ個別にコピーする方が安定する。
- 2026-03-08: 左固定列が複数ある表では、2列目の `left` を固定値にしない。1列目の実測幅を CSS 変数へ流して追従させないと、横向きやSafariで数pxずれやすい。
- 2026-03-08: 左固定列を含む固定ヘッダー全体を `transform` で横移動すると、固定列まで一緒に流れる。別ラッパー式のヘッダーは `scrollLeft` 同期にして、sticky 列の計算を壊さない。
- 2026-03-08: 薬の即答用カードでは、良い組み合わせだけでなく NG も閉じた状態で見える方が判断が速い。ただし NG は常時出さず、存在するときだけ2段目に出すと一覧性を崩しにくい。
- 2026-03-09: 公開前の信頼表示はカード内バッジだけでは足りない。ヘッダー直下の免責と、全公開ページ共通の「更新日・出典・免責」フッターを先に揃えると、PWA公開時の説明不足を減らしやすい。
- 2026-03-09: 公開ルールの説明は、公開件数や非公開内訳まで出すと運用情報が前面に出すぎることがある。公開条件だけを短く明示した方が、一般ユーザーには読みやすい。
- 2026-03-09: 静的PWA化で `index.html` へ寄せるときは、HTMLのリネームだけで終わらせない。テストやローカル検証コマンド、manifest/SW 参照先まで同時に切り替えないと確認系が壊れる。
- 2026-03-09: PWA のプレースホルダーアイコンを空ファイルで置く場合、実装上は先に進めても、インストール性や Lighthouse の最終確認は実画像配置後にやり直す前提を明記する。
- 2026-03-10: iPhone の下部ジェスチャー領域に固定ナビをベタ付けすると、中央タブでもホーム操作に吸われやすい。実機で誤タップが出たら、ナビ全体を少し浮かせて内側余白を増やす方が効く。
- 2026-03-10: 公開データのカテゴリ不具合はフィルター関数より先にデータ分類を疑う。`category` 配列の誤りは一覧・表・タブ全体へ波及するので、根本修正はデータ側で行う。
- 2026-03-10: PWA の表示名は `<title>` だけでは揃わない。iPhone のホーム画面名まで統一したいなら `apple-mobile-web-app-title` と manifest の `short_name` も同じ文言に揃える。
- 2026-03-10: iOS の「ホーム画面に追加」では、アイコン下の表示名と入力欄の初期値が別ソースになることがある。ずれを減らすには `application-name` と実行時 `document.title` も同じ名称へ寄せる。
- 2026-03-10: 複数薬まとめは上位タブを増やす前に、飲み合わせ表のモード切替として入れる方が薬剤師の導線に合う。新タブ化より「個別表の延長」として理解しやすい。
- 2026-03-10: 複数薬の共通候補は、元データの `mixGood` を直接突き合わせるより `getCompatEntries()` の出力で集計した方が、公式資料由来データと追加薬データを同じルールで扱える。
- 2026-03-10: 複数薬まとめUIは、集計ロジックだけ正しくても見つけにくいと使われない。モード切替は目立たせ、選択済み薬は個別解除できるようにし、補足文は初期状態で折りたたんだ方が現場向き。
- 2026-03-10: 医療情報UIのラベルは、意味が重複する接頭辞を繰り返さない方が読みやすい。`資料上:` のような前置きは凡例やセクション名で担保し、各行の文言は短く保つ。
- 2026-03-10: 新機能のモード切替は、機能追加後に「存在に気づかれない」ことがある。重要な切替は周辺UIと役割を揃えた色へ寄せて、視線が先に届くようにした方が使われやすい。
- 2026-03-10: 2択モードの切替は、単なる並列ボタンよりセグメント+スライドサムの方が「今どちらのモードか」が一瞬で分かる。薬剤師向けの即答UIでは状態認識を優先する。
- 2026-03-10: 複数薬まとめの候補は1列で見せると縦に伸びすぎる。詳細を折りたたむ前提なら、食品カードは3列程度まで詰めた方が一画面で比較しやすい。
- 2026-03-10: 複数薬まとめの主操作は、表内チェックより上部のチップ選択に寄せた方がスマホで押しやすい。薬剤師の作業は「薬を選ぶ → 共通候補を見る」であり、行列確認は二次操作に下げる方が自然。
- 2026-03-10: 主操作を別UIへ移した後の表は、選択手段を二重化しない方が迷いが少ない。確認用の表には `選択中` 表示だけ残し、責務を分けた方が理解しやすい。
- 2026-03-10: 別レイヤーの sticky ヘッダーは、初回描画直後の1回同期だけでは高さ変数がずれることがある。横スクロールで自然回復する不具合が出たら、描画後の `requestAnimationFrame` 再同期を追加して潰す。
- 2026-03-10: モード切替の絵文字は機能の意味と揃える。複数薬まとめでは試験管より薬そのものを連想できる記号の方が、現場の理解が速い。
- 2026-03-10: 選択UIと結果UIが同じトーンで連続すると、更新されても結果に気づかれにくい。主操作と結果は別パネルに分け、結果側には常設の空状態と更新フィードバックを置いた方がよい。
- 2026-03-10: 確認用の個別表に残す `選択中` 表示は、ただの文字だけだと埋もれやすい。色・枠・小アイコンを揃えたバッジにすると、スクロール中でも視認しやすい。
- 2026-03-11: 患者向けの文脈を足したい時は、レイアウトを変える前に見出しと言い回しを `今回の処方` ベースへ寄せる。薬剤師向けの構造を保ったまま、自分ごと感だけを足しやすい。
